# 迁移与架构回滚方案

## 回滚单位

- 代码：按堆叠 PR 回滚，不跨阶段打包回滚。
- 数据库：所有迁移向前追加；不通过降级 SQL 删除新表或新列。
- 正文：以单部作品为最小迁移和回滚单位。
- 本地草稿：保留旧 IndexedDB 和来源版本，兼容期内不自动删除。

## 逐作品回滚

1. 校验迁移前备份哈希和 `migration_id`。
2. 暂停该作品的新格式写入并完成当前本地草稿持久化。
3. 保存新格式回滚前快照，不删除新数据。
4. 将读取指针切回对应旧格式来源。
5. 验证卷章数、章节标题、正文哈希、版本数和权限成员。
6. 标记迁移运行 `rolled_back` 并记录原因、操作者和时间。

## 发布回滚

- PR 0 与后续每层 PR 保持可独立撤销；不得在同一提交混入跨阶段数据库清理。
- 前后端版本不一致时只提示更新，不在写作中强制刷新。
- 新 API 客户端在兼容期识别旧响应；无法兼容时显示恢复入口，不循环重试。
- Service Worker 只缓存版本化公开静态资源；回滚后激活新静态版本，但不得清理私人 IndexedDB。

## 禁止操作

- 不执行 `DROP TABLE`、清空正式 D1、删除旧对象或重建用户 IndexedDB。
- 不把迁移失败解释为可以覆盖来源数据。
- 不在没有备份恢复演练的情况下结束双读兼容期。
