# 本地草稿加密与密钥生命周期审计

## 密钥边界

- 每个用户生成独立随机 32 字节 DEK。
- Worker Secret 提供根密钥材料；KEK 必须按用户 ID、用途和版本派生，避免所有用户直接共享同一封装密钥。
- D1 `user_local_draft_keys` 只保存 `wrapped_dek`、`wrap_iv`、`kek_version` 和时间，不保存明文 DEK。
- 浏览器完成认证后通过受保护、`no-store` 的同源接口取得解封结果；DEK 只保留在内存。
- DEK 不写入 IndexedDB、localStorage、sessionStorage、Cookie、Service Worker 或日志。

## 生命周期

1. 登录并验证会话。
2. 服务端按用户派生 KEK，解封或首次生成 DEK。
3. 浏览器仅在当前认证会话的内存中保存 DEK。
4. 草稿、同步队列、设置和冲突草稿按用户命名空间使用 AES-GCM 加密。
5. 注销、会话撤销、锁定、账号切换、密钥错误和页面安全退出时清除内存引用。
6. 同一用户重新登录后才能再次解封；其他账号不能枚举或读取该命名空间。

## 失败关闭

- 缺少根密钥、封装记录损坏、版本不支持或解封失败时拒绝读写草稿。
- 不得退化为明文，不得生成新 DEK 覆盖无法解封的旧 DEK。
- 恢复界面只允许导出仍可解密的数据；无法解密时保留密文并提示管理员恢复正确密钥版本。

## 与 PR 0 的结合

加密层必须接入 PR 0 的 IndexedDB 状态机，处理 blocked、versionchange、多标签升级和只读恢复。数据库升级失败不得删除密文或自动重建数据库。
